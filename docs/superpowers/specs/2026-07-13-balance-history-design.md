# Balance History (AC3) — Design

**Date**: 2026-07-13
**Status**: approved pending user review
**Source requirement**: `docs/moneymanager-feature-reference.md` AC3 (the node the user linked on the Figma board): balance-over-time line chart. User decisions 2026-07-13: absolute balances via **one-shot EB calibration** (not manual entry, not relative-only); chart lives in **AccountsPage**.

## Goal

Show the real total balance over time (monthly line chart) in AccountsPage, and make the accounts list / net worth absolute instead of delta-based. Absolute truth comes from a one-time calibration against Enable Banking's balances endpoint; everything after that is transaction math.

## Requirements

1. New `accounts` table storing per-account opening balances (schema via Alembic — first revision after baseline).
2. One-shot calibration script: read current real balance from EB, derive `opening_balance = eb_balance − SUM(all deltas)`, upsert. Re-runnable on demand; never scheduled (PSD2 rate-limit prudence).
3. `GET /v1/stats/balance-history?months=N` returning a monthly cumulative series of the total balance.
4. AccountsPage: new "Balance" line-chart section + existing list/net-worth corrected by opening balances.
5. All existing invariants hold: Alembic-only DDL, `/v1` envelope, JWT guard, float cast at the service boundary, `isAnimationActive={false}` on new recharts series, taxonomy untouched.

## Architecture

### Schema — Alembic revision 0002

```sql
CREATE TABLE accounts (
    account_uid     TEXT PRIMARY KEY,          -- matches transactions.account_id
    display_name    TEXT,                      -- optional human name (UI falls back to uid)
    opening_balance NUMERIC NOT NULL DEFAULT 0,
    eb_balance      NUMERIC,                   -- EB-reported balance at calibration (audit)
    calibrated_at   TIMESTAMPTZ                -- when calibration last ran
);
```

No FK to `transactions.account_id` (kept TEXT/loose, consistent with current style). Autogenerated revision, manually reviewed (repo rule).

### EB client — `ingestion/fetch_transactions.py`

New `fetch_balances(client, account_uid) -> Decimal` reusing the existing `_get`/JWT plumbing on `GET /accounts/{uid}/balances`. Response is Berlin Group-style `{"balances": [{"balance_amount": {"currency", "amount"}, "balance_type": ...}]}`: prefer `balance_type == "CLBD"` (closing booked), else the first entry. Log a warning if currency ≠ EUR and proceed (declared assumption: EUR accounts, consistent with the app-wide `eur_amount` convention).

### Calibration — `scripts/calibrate_balances.py`

Pattern of `migrate_taxonomy.py` (run via `railway run --service just-comfort`). For each uid in the configured EB account ids: fetch EB balance, `SELECT COALESCE(SUM(eur_amount), 0) FROM transactions WHERE account_id = %s` — **all rows, including `is_internal=TRUE`**: top-ups/vault moves change the real balance; excluding them would mis-calibrate. Then upsert:
`INSERT ... ON CONFLICT (account_uid) DO UPDATE SET opening_balance/eb_balance/calibrated_at`.
2s inter-account delay (existing convention). Logs one line per account (uid, eb_balance, delta_sum, opening).

### Service — `services/stats.py::balance_history(conn, months=12)`

1. `openings = SELECT COALESCE(SUM(opening_balance), 0) FROM accounts`
2. Monthly nets: `SELECT TO_CHAR(DATE_TRUNC('month', booking_date), 'YYYY-MM'), SUM(eur_amount) FROM transactions WHERE account_id IS NOT NULL GROUP BY 1` — internal rows **included** (same reasoning as calibration); `account_id IS NULL` (manual) rows **excluded**, consistent with today's accounts page scope.
3. Python assembles a continuous month axis from `min(first data month, now − months + 1)` through the current month, cumulative-sums onto `openings`, carries the value forward through empty months, slices the last `months`, casts to `float`.

Returns ascending `[{"month": "YYYY-MM", "balance": float}]`.

### API

`GET /v1/stats/balance-history` with `months: MonthsQ = 12` (existing 1..24 annotated type), JWT via `router_v1`, `{"data": [...]}` envelope, `-> dict` style.

### Accounts coherence (same truth everywhere)

`services/accounts.py::balances` gains a LEFT JOIN on `accounts`: per-account `balance = opening_balance + SUM(eur_amount)` and `display_name` in the payload (frontend renders `display_name ?? account_id`). Assets/liabilities math unchanged — numbers just become absolute. Uncalibrated accounts behave exactly as today (opening 0).

### Frontend

- `types.ts`: `BalancePoint { month: string; balance: number }`; `AccountBalance` gains `display_name: string | null`.
- `client.ts` / `queries.ts`: `api.stats.balanceHistory({months})`, `statsQueries.balanceHistory(months = 12)`.
- `AccountsPage.tsx`: "Balance" section between hero and list — recharts `LineChart` (monthly X, € tooltip, `isAnimationActive={false}` per the background-tab lesson), local `formatMonth` helper (2nd usage in codebase; rule-of-three not met, extraction deferred).

## Declared limits

- History starts at the first synced month; earlier window points render flat at the openings total. EB provides no historical balances.
- Single-currency assumption (EUR). Multi-currency accounts would need per-currency openings — out of scope.
- Total-only series (no per-account lines).

## Testing

- Service: cumulative math with mocked rows (openings + monthly nets), carry-forward across empty months, window slicing, floats.
- API: 401 unauth; 200 shape with month strings + float balances; months validation 422.
- Calibration: mocked EB response + cursor — correct opening math, CLBD preference, upsert SQL/params.
- `fetch_balances`: CLBD picked over others; fallback to first; non-EUR warning.
- Frontend: AccountsPage renders the section from mocked query (existing test patterns); line points present in DOM.
- Migration: autogenerate reviewed by hand; applied to prod via `railway run -- alembic upgrade head` in the ops task.

## Out of scope

- Per-account balance series, scheduled recalibration, non-EUR support, backfilled pre-history, MoneyManager-style bar chart pairing (exists in StatsPage already).

## Decisions log

- Absolute balances via one-shot EB calibration (user choice 2026-07-13); manual entry remains possible by editing `accounts.opening_balance` directly.
- Chart placement: AccountsPage (user choice 2026-07-13), matching the board's 📊→Total Stats arrow.
- Internal transactions count toward balance math (calibration and series) — they move real money; `real_transactions` filtering stays a stats-page concern only.
- Manual (`account_id IS NULL`) rows excluded from the series, mirroring the accounts page.
